## Challenges & Resolutions

### 1. LIS TypeScript Build Errors (25 errors)
**Symptom:** Cloud-init build of rxsoft-lis-backend failed with 25 TS errors.
**Root Cause:** 18 service methods declared `tenant?: TenantContext` param but it didn't exist in method signature. Two controllers had `transitionStatus()` arguments swapped (`tenantRole`, `tenantContext` vs `tenantContext`, `tenantRole`).
**Resolution:** Added optional `tenant?: TenantContext` param to all 18 service methods in OrdersService, ResultsService, SpecimensService, etc. Fixed argument order in `orders.controller.ts` and `results.controller.ts`.
**Prevention:** Add TS strict type-checking pre-commit hook for rxsoft-lis-backend.

### 2. ehealthwares Next.js HOSTNAME binding
**Symptom:** Container showed `unhealthy` â€” healthcheck `node -e http.get('http://localhost:3000/')` returned ECONNREFUSED.
**Root Cause:** Next.js standalone `server.js` uses `process.env.HOSTNAME || '0.0.0.0'` for binding. Docker automatically sets `HOSTNAME` to container ID, so the server bound to `172.18.0.7:3000` (container IP) instead of `0.0.0.0:3000`. This meant `localhost:3000` was not listening.
**Resolution:** Added `HOSTNAME: "0.0.0.0"` to ehealthwares service environment in `docker-compose.prod.yml` to override Docker's automatic HOSTNAME.
**Prevention:** Document in Dockerfile that Next.js standalone needs `HOSTNAME=0.0.0.0` in Docker environments.

### 3. nginx rxsoft.conf empty server_name
**Symptom:** `dns-update.sh --env prod` generated `server_name ;` (empty), causing nginx reload to fail with `invalid number of arguments in "server_name"`.
**Root Cause:** The awk YAML parsing in `dns-update.sh` line 55-57 is broken for YAML format where server_names are on separate lines (not inline). The pattern `/frontend:/` matched, but `for(i=2;i<=NF;i++)` iterated over fields on the `server_names:` line, which has no inline values.
**Resolution:** Manually wrote correct `rxsoft.conf` and `api.conf` with proper `server_name rxsoft.ehealthwares.com damorex.ehealthwares.com apm.ehealthwares.com conversation.ehealthwares.com;` and deployed via docker cp.
**Prevention:** Fix `dns-update.sh` YAML parsing to read multi-line `- subdomain` entries instead of expecting inline array.

### 4. ehealthwares tarball deployment (empty GitHub repo)
**Symptom:** `post-deploy.sh` LIS build failed because GitHub `rxsoft-lis-backend` repo had stale Dockerfile without new TS fixes.
**Resolution:** SCP'd full source tarball of ehealthwares directly to server and built locally.
**Prevention:** Push all fixes (Dockerfiles, TS fixes, etc.) to GitHub repos and rebuild base images.

### 5. rxsoft-admin dependency on ehealthwares container
**Symptom:** Admin nginx failed to start because `upstream rxsoft-ehealthwares` was defined in default.conf and nginx resolves upstream names at config load time.
**Resolution:** Used variable-based `proxy_pass $ehealthwares_upstream;` with `set` + `resolver` for deferred DNS lookup.
**Prevention:** Already in place via nginx-default.conf variable pattern.
