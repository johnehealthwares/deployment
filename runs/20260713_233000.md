## Issues Encountered & Resolutions

### 1. ECR repos missing from terraform state

**Symptom:** `terraform apply` failed with `RepositoryAlreadyExistsException` for rxsoft-ehealthwares and 3 other repos.
**Root Cause:** `teardown.sh --keep-s3-ecr` ran `terraform state rm` on ECR repos before destroying, so terraform state lost track of them while resources still existed in AWS.
**Resolution:** Ran `terraform import` for all 8 ECR repos. Lifecycle policies didn't exist in AWS (also removed), so they were created fresh by `terraform apply`.
**Prevention:** Added recovery import commands as a comment block in `terraform/cloud-init.sh` header.

### 2. Cloud-init didn't re-run on instance modification

**Symptom:** After `terraform apply`, the EC2 instance was modified in-place (user_data change only), but `/home/ubuntu/develop/` was empty and no containers were running.
**Root Cause:** Cloud-init modules run `once-per-instance`. A user_data change on an existing instance only triggers modules with `frequency=always` — the bootstrap script (config-scripts_user) is `once-per-instance`.
**Resolution:** Manually ran the bootstrap steps: clone repos, setup keyfile/init scripts, ECR auth, docker compose pull + up.
**Prevention:** When instance needs reprovisioning, use `terraform taint aws_instance.postgres` then `terraform apply` to force recreation.

### 3. Broken nginx default.conf from GitHub deployment repo

**Symptom:** `nginx -s reload` failed with "invalid number of arguments in 'set' directive in /etc/nginx/conf.d/default.conf:15".
**Root Cause:** The GitHub deployment repo's `docker/nginx-default.conf` has malformed `set` directives — missing variable names: `set http://rxsoft-backend:8080/api/;` instead of `set $u_backend http://rxsoft-backend:8080/api/;`. Cloud-init.sh was supposed to overwrite this, but cloud-init didn't re-run (see issue #2).
**Resolution:** Wrote correct nginx config directly into the admin container via `docker exec -i rxsoft-admin tee`.
**Prevention:** Push the fixed `nginx-default.conf` to the GitHub deployment repo so fresh git clones get the correct version.

### 4. ehealthwares port 8083 not open in security group

**Symptom:** `curl http://52.18.107.253:8083/` returned `000` (connection refused).
**Root Cause:** Security group had ingress for ports 8080-8082 but not 8083. The ehealthwares compose service maps to host port 8083.
**Resolution:** Extended security group ingress range from 8080-8082 to 8080-8083 in `terraform/main.tf` and applied.
**Prevention:** Port range now covers ehealthwares. When adding new services, verify SG ports.
