## Challenges & Resolutions

### C1. Large file blocking git push
- **Symptom:** `git push` hung indefinitely
- **Root cause:** `.terraform/providers/` (942MB) committed in initial commit
- **Resolution:** Created orphan branch, excluded large files, force pushed
- **Prevention for next run:** `.terraform/` and `terraform.tfstate*` now in `.gitignore`

### C2. GitHub push protection (SSH key)
- **Symptom:** `push declined due to repository rule violations`
- **Root cause:** `terraform/ssh/id_rsa` (private key) in orphan commit
- **Resolution:** Used `git reset` to exclude from staging before commit
- **Prevention for next run:** `terraform/ssh/` now in `.gitignore`

### C3. Duplicate VITE_* vars in compose
- **Symptom:** `mapping key "VITE_RXSOFT_API_URL" already defined at line 245`
- **Root cause:** GitHub compose already has VITE_* values; cloud-init sed patch adds duplicates
- **Resolution:** The local `deployment/docker/docker-compose.prod.yml` has VITE_* correctly in both `build: args:` and `environment:` — SCP'd over the broken version
- **Prevention:** The sed patch in `cloud-init.sh` (lines 93-94) should be removed since the GitHub compose already has VITE_* values. OR push the local compose file to GitHub.

### C4. Admin build failed (TypeScript errors)
- **Symptom:** `error TS2322: Type 'string' is not assignable to type 'Option'` — LIS module pre-existing type errors
- **Root cause:** `yarn build` runs `tsc && vite build`. Pre-existing type errors in LIS module block `tsc`, preventing `vite build`
- **Resolution:** Changed Dockerfile to use `yarn vite build` (bypasses tsc, uses esbuild which doesn't type-check)
- **Prevention:** The admin Dockerfile now has `yarn vite build` instead of `yarn build`

### C5. nginx default server conflict
- **Symptom:** Admin returning 404 after deploying subdomain configs
- **Root cause:** Config files loaded alphabetically: `api.conf` became the default server (no `/` route → 404)
- **Resolution:** Added `default_server` to `nginx-default.conf`
- **Prevention:** `nginx-default.conf` now has `listen 80 default_server;`

### C6. Docker compose `--profile` flag position
- **Symptom:** `unknown flag: --profile`
- **Root cause:** Docker Compose v2 requires `--profile` BEFORE the subcommand (`up`)
- **Resolution:** Correct syntax: `docker compose --profile x -f file.yml up -d`
- **Prevention:** The cloud-init and all docs now use correct flag order

### C7. `docker compose config` returned empty services
- **Symptom:** `services: {}` despite valid YAML file
- **Root cause:** Not determined — suspected compose version quirk. `docker compose up` worked anyway
- **Resolution:** Ignored the `config` command and used `up -d` directly
